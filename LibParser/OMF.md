# Формат OMF и библиотек .LIB (Microsoft QuickC)

Документ описывает формат, который разбирает **LibParser**: OMF-библиотеки (`.LIB`) 16-битного линкера Microsoft LINK / LIB для QuickC, MASM 5.x и близких инструментов.

> **Не путать с COFF.** Файлы с расширением `.LIB` бывают двух несовместимых видов:
> - **OMF** — архив объектных модулей OMF (QuickC, старый MSC, MASM для DOS). Начинается с байта `F0h`.
> - **COFF** — архив с сигнатурой `!<arch>` (32/64-bit Visual C++).
>
> UltraDecompiler и LibParser работают только с **OMF**.

Первоисточник: *Tool Interface Standards (TIS), Relocatable Object Module Format (OMF), Version 1.1* — Appendix 2 (Library File Format). Краткая выжимка ниже согласована с реализацией в `LibParser/Omf/`.

---

## Общая структура OMF-записи

Любая запись в объектном модуле (и заголовок библиотеки в форме записи) имеет вид:

```
┌──────────┬────────────────┬─────────────────────────┐
│ Type (1) │ Length (2, LE) │ Содержимое (Length байт)│
└──────────┴────────────────┴─────────────────────────┘
```

| Поле | Описание |
|------|----------|
| **Type** | Тип записи (например `80h` = THEADR, `A0h` = LEDATA). |
| **Length** | Длина **остатка** записи (без Type и Length), **включая** последний байт контрольной суммы. |
| **Содержимое** | Полезные данные + **1 байт checksum** в конце (часто `00h`; линкер принимает и нулевую «сумму»). |

Размер записи в файле: `3 + Length` байт.

**16/32-битные поля внутри записи:** если младший бит типа записи равен 1 (нечётный тип, например `A1h`, `99h`), многие числовые поля в этой записи — 4 байта; иначе — 2 байта (для Use32 сегментов значения нулевым словом дополняются до 32 бит).

**Индексы OMF:** ссылка на предыдущий SEGDEF, имя в LNAMES и т.д. кодируется 1 или 2 байтами. Если старший бит первого байта = 0 — индекс в одном байте; иначе — `(byte0 & 7Fh) << 8 | byte1`.

**Строки:** обычно «счётчик + ASCII», без завершающего нуля: `Len (1)` + `символы (Len)`.

## Структура файла OMF-библиотеки (.LIB)

Файл на диске — кратность **512 байт** (сектор DOS). Логически:

```
┌─────────────────────────────────────────────────────────────┐
│ Страница 0: заголовок библиотеки (запись F0h)               │
├─────────────────────────────────────────────────────────────┤
│ Страница 1..N: объектные модули (OMF, LHEADR/THEADR..MODEND)│
│   (каждый модуль выровнен по размеру «страницы» библиотеки) │
├─────────────────────────────────────────────────────────────┤
│ Запись F1h (конец модулей, выравнивание до 512)             │
├─────────────────────────────────────────────────────────────┤
│ Словарь: DictionaryBlockCount × 512 байт                    │
├─────────────────────────────────────────────────────────────┤
│ (опционально) расширенный словарь F2h — зависимости модулей │
└─────────────────────────────────────────────────────────────┘
```

### Заголовок библиотеки (F0h)

Первая запись файла; тип `F0h`. **Не** использует checksum по правилам обычных OMF-записей.

| Смещение в записи | Размер | Поле |
|-------------------|--------|------|
| +0 | 1 | `F0h` |
| +1 | 2 | Record Length (LE) → **PageSize = Length + 3** |
| +3 | 4 | Смещение первого байта словаря (LE) |
| +7 | 2 | Число 512-байтных блоков словаря |
| +9 | 1 | Флаги (`01h` = регистрозависимый поиск) |
| остальное | — | padding (не используется) |

Заголовок занимает ровно **одну страницу** (`PageSize` байт). У QuickC в `CLIBC.LIB` обычно `PageSize = 16` (Record Length = `0Dh`).

**Номер страницы** модуля в словаре:

```
PageNumber = FileOffset_первой_записи_модуля / PageSize
```

Страница **0** — заголовок F0; первый модуль чаще всего на странице **1** (смещение `0x10` при PageSize 16).

### Объектный модуль внутри библиотеки

Каждый модуль — обычный OMF object module:

1. **THEADR (`80h`)** или **LHEADR (`82h`)** — имя (часто исходный файл, например `printf.c`).
2. **COMENT (`88h`)**, класс **A3h (LIBMOD)** — короткое имя модуля в библиотеке (базовое имя `.OBJ`, например `printf`).
3. **LNAMES (`96h`)** — таблица имён сегментов и классов (`CODE`, `DATA`, `_TEXT`, …).
4. **SEGDEF (`98h`/`99h`)** — описание сегментов; индексы с 1.
5. **PUBDEF, EXTDEF, …** — символы (линкер; LibParser для извлечения кода не разбирает детально).
6. **LEDATA (`A0h`)** / **LIDATA (`A2h`)** — инициализированные данные и **код** функций.
7. **FIXUPP (`9Ch`)** — релокации (для сопоставления с EXE нужны линкеру; в LibParser сохраняются в `RawData`).
8. **MODEND (`8Ah`/`8Bh`)** — конец модуля.

После MODEND модуль дополняется нулями до границы **страницы библиотеки** (`PageSize`).

### Конец объектных модулей (F1h)

Запись `F1h` отделяет модули от словаря; длина подобрана так, чтобы словарь начинался на границе **512 байт** (и совпадала с полем «смещение словаря» в F0).

## Записи, важные для извлечения кода

### THEADR / LHEADR (80h / 82h)

```
StringLen (1) | Name (StringLen) | Checksum (1)
```

В библиотеках чаще **LHEADR**; THEADR встречается в старых модулях (`87.LIB`).

### COMENT — LIBMOD (88h, класс A3h)

Только в `.LIB`; линкер игнорирует.

```
CommentType (1) | CommentClass (1) | … | A3h | Len (1) | ModuleName (Len) | …
```

У QuickC класс `A3h` иногда идёт после двух нулевых байт в теле COMENT.

### LNAMES (96h)

Последовательность строк `Len + ASCII`; индекс **0** зарезервирован (пустое имя). Дальнейшие SEGDEF ссылаются на индексы в объединённом списке имён модуля.

### SEGDEF (98h / 99h)

После типа и длины:

- **ACBP** (1 байт): Alignment, Combination, Big, Use16/32.
- При Alignment = 0 — дополнительно Frame + Offset (3 байта).
- **Segment Length** (2 или 4 байта).
- **Segment Name Index**, **Class Name Index**, **Overlay Index** (OMF-индексы).

Код QuickC обычно в сегменте с классом **`CODE`** (имя часто `_TEXT` или `имяфайла_TEXT`).

### LEDATA (A0h / A1h)

```
SegmentIndex | DataOffset (2 или 4) | Data… | Checksum
```

Сырые байты кода/данных по смещению в сегменте. LibParser склеивает несколько LEDATA одного сегмента в один массив `OmfSegmentData.Data`.

### LIDATA (A2h / A3h)

Итеративные (повторяющиеся) данные — типичны для инициализированных массивов в C. Структура:

```
SegmentIndex | Offset | RepeatCount | BlockCount | DataBlocks…
```

**Data block:** если `BlockCount = 0` — `ByteCount` + байты, повторённые `RepeatCount` раз; иначе вложенные блоки. LibParser разворачивает LIDATA в плоский массив (см. `OmfLidataExpander`).

### EXTDEF (8Ch)

Список внешних символов модуля (ссылки на другие `.OBJ` / модули библиотеки):

```
StringLen (1) | Name (Len) | TypeIndex (1–2) | …
```

Индексы EXTDEF (совместно с LEXTDEF/COMDEF) используются в FIXUPP как **Target Datum** (1-based). LibParser сохраняет их в `OmfModule.ExternalSymbols`.

### FIXUPP (9Ch / 9Dh)

Релокации для **непосредственно предшествующей** LEDATA/LIDATA. Запись состоит из подзаписей:

| Старший бит 1-го байта | Тип |
|------------------------|-----|
| 0 | **THREAD** — задаёт FRAME/TARGET thread 0…3 для следующих FIXUP |
| 1 | **FIXUP** — одна релокация |

#### THREAD (1 байт + опциональный индекс)

```
0 D 0 Method Thread | [Index]
```

- **D=0** — TARGET thread, **D=1** — FRAME thread.
- **Method** — 2 бита (TARGET) или 3 бита (FRAME): 0=SEGDEF, 1=GRPDEF, 2=EXTDEF, …
- **Thread** — номер 0…3.
- **Index** — если Method ≤ 2.

#### FIXUP (Locat 2 + Fix Data 1 + …)

**Locat** (16 бит, старшие биты в младшем байте):

```
byte0: 1 | M | Location[3:0]
byte1: DataRecordOffset[9:0]  (смещение в LEDATA/LIDATA)
```

- **M=1** — segment-relative, **M=0** — self-relative (PC-relative).
- **Location** — тип поля: 0=byte, 1=word/offset16, 9=offset32, …

**Fix Data** (1 байт):

```
F | Frame[2:0] | T | P | Target[1:0]
```

- **F=1** — FRAME из thread; **F=0** — явный метод Frame[2:0] (F0…F5).
- **T=1** — TARGET из thread; **T=0** — явный (P+Target как 3-битный метод T0…T6).
- **P=1** — без Target Displacement (T4/T5/T6); **P=0** — следом 2 или 4 байта смещения (9Ch/9Dh).

**Frame Datum / Target Datum** — OMF-индекс SEGDEF, GRPDEF или EXTDEF (если метод 0–2).

LibParser сохраняет разобранные FIXUP в `OmfModule.Fixups` (`OmfFixup`: сегмент, смещение, FRAME, TARGET с разрешёнными именами EXTDEF/SEGDEF).

## Словарь публичных символов

Словарь — `DictionaryBlockCount` блоков по **512 байт** каждый, начиная с `DictionaryOffset` из заголовка F0.

### Макет одного блока (512 байт)

| Байты | Назначение |
|-------|------------|
| 0–36 | **HTAB** — 37 «корзин» хеш-таблицы: значение `B` → смещение записи `2×B` в блоке |
| 37 | **FFLAG** — индекс свободного места (`×2` = следующий свободный байт); `FFh` = блок полон |
| 38 … | Записи символов (упакованы) |

### Формат записи символа

```
Len (1) | Name (Len) | ModulePage (2, LE) | [pad 1 байт если нужно выровнять]
```

- **ModulePage** — номер страницы библиотеки, с которой начинается модуль-носитель символа (см. выше).
- Имена выровнены так, чтобы следующая запись начиналась на чётном смещении.

Поиск в LINK использует двухуровневое хеширование (номер блока + корзина 0…36, шаги `block_d` / `bucket_d`). LibParser для **перечисления** всех символов сканирует область данных блока и корзины HTAB (см. `OmfDictionaryParser`).

**Пример (`87.LIB`):**

| Символ | ModulePage | Смещение модуля (PageSize=16) |
|--------|------------|-------------------------------|
| `__fpmath` | 1 | `0x10` |
| `__FPINSTALL87` | 173 | `0xAD0` |

## Расширенный словарь (F2h, опционально)

После основного словаря (LIB ≥ 3.09) может идти:

```
F2h | Length (2) | N modules | таблица (Page, offset) | списки зависимостей…
```

Описывает, какие модули нужно подтянуть вместе при линковке одного символа. LibParser пока **не разбирает** extended dictionary.

## Соответствие API LibParser

| Концепция OMF | Тип в коде |
|---------------|------------|
| Весь файл | `OmfLibrary.RawData` |
| Заголовок F0 | `OmfLibraryHeader` |
| Модуль | `OmfModule` (`PageNumber`, `FileOffset`, `HeaderName`, `LibraryModuleName`, `Segments`, `RawData`) |
| Сегмент с данными | `OmfSegmentData` (`SegmentName`, `ClassName`, `Data`; `IsCode` ⇔ класс `CODE`) |
| EXTDEF | `OmfExternalSymbol` (`Index`, `Name`) |
| FIXUP | `OmfFixup` (`SegmentOffset`, `LocationType`, `Frame`, `Target`) |
| Запись словаря | `OmfPublicSymbol` (`Name`, `ModulePage`) |
| Поиск | `FindModuleBySymbol` → `GetModuleByPage(ModulePage)` |

```csharp
var lib = OmfLibraryParser.ParseFile("CLIBC.LIB");
var sym = lib.Symbols["_printf"];           // есть в словаре
var mod = lib.FindModuleBySymbol("_printf"); // mod.PageNumber == sym.ModulePage
var code = mod?.CodeSegments.FirstOrDefault()?.Data;
```

## Отличия от COFF .LIB (кратко)

| | OMF (QuickC) | COFF (Visual C++) |
|---|--------------|-------------------|
| Начало файла | `F0h` | `!<arch>` |
| Элемент | OMF module (записи 80h…) | COFF .obj + заголовок архива |
| Таблица символов | Хеш-словарь в конце | Два linker member в начале |
| Релокации | FIXUPP, сегменты 16-бит | COFF relocations |
